## 2. 방법

앱의 실제 파이프라인(`recognizeReceiptWithPpOcr`)과 **동일한 로직**을 Node에서 재현하여 각 이미지를 인식했다.

1. **검출 전처리** — 원본을 최대변 960px로 축소(32 배수 정렬), ImageNet 정규화, NCHW float32 텐서.
2. **텍스트 검출** — DB(Differentiable Binarization) 후처리: 확률 임계값 `0.3`, boxThreshold `0.5`, minArea `12`, unclip `1.6`, 연결요소 → 축 정렬 박스 → 가로행 병합(수직 겹침 ≥0.45).
3. **인식 전처리** — 각 박스를 원본에서 crop, 높이 48·폭≤320으로 리사이즈, `/127.5 − 1` 정규화.
4. **텍스트 인식** — 인식 모델 추론 후 **CTC 그리디 디코딩**(사전 매핑), 신뢰도 = 스텝별 최대 로짓 평균.
5. **필터** — 신뢰도 `≥ 0.35` 인 줄만 채택.

**유일한 근사:** 이미지 리사이즈/크롭이 앱에서는 iOS 네이티브(`expo-image-manipulator`), 본 테스트에서는 bilinear로 구현됨. 인식 결과에 미치는 영향은 미미.

## 5. 핵심 발견

> **[가장 중요한 맥락] 오늘 측정한 PP-OCR이 곧 “현재 iOS에서 실제로 도는 엔진”이다.**
> 코드상 iOS 인식 우선순위는 **Apple Vision → CLOVA → PP-OCR** 이지만,
> `appleVisionRecognizer`는 `"native bridge is not connected yet"`, `clovaRecognizer`는
> `"HTTP adapter is not implemented yet"` 로 **둘 다 스텁(throw)** 상태다
> (`app/platform/receiptRecognizers.ts`). 따라서 `IOS_INTERIM_PPOCR_FALLBACK = true` 로
> **PP-OCRv5가 임시 대타로 실행**된다. 즉 R01·R04·R05의 붕괴는 “원래 주력이어야 할 엔진이
> 아직 연결되지 않았기 때문”이며, 이것이 아래 개선방향의 1순위를 규정한다.

1. **검출 안정성 편차가 큼.** 모델 자체의 한글 인식력은 준수하나(R07은 26줄 중 전화·상품명·배달장소·받는분·보내는분까지 대부분 포착), 검출 단계가 영수증마다 불안정하여 R01·R04·R05에서는 본문 대부분을 놓쳤다.
2. **기울기 보정(deskew)이 없음.** 검출 박스가 **축 정렬(axis-aligned)** 이고 회전 crop이 없어, R04처럼 크게 기울어진 영수증은 글자가 사선으로 잘려 인식이 붕괴한다. 실사용에서 기사님이 비스듬히 촬영하면 그대로 실패로 이어진다.
3. **가로행 병합이 과함.** 한 가로줄을 넓게 묶어(수직 겹침 기준) 서로 다른 표 칸(예: “품명 / 축하3단 / 수량 / 1”)이 하나의 crop에 섞여 들어가 오인식을 유발한다.
4. **선명해도 놓치는 케이스(R01).** 사람 눈에는 매우 선명한데도 검출이 4영역만 잡고 로고 그래픽을 오독했다 — 단일 960 스케일 + DB 임계값 조합의 한계로 보인다.
5. **표 라벨(품명·수량·TEL·배달장소 등) 인식은 강함.** 고정 인쇄 라벨은 신뢰도 0.9~1.0으로 안정적 → 필드 앵커로 활용 여지.

## 6. 앞으로의 개선방향

측정 결과와 코드 구조(5절 맥락)를 종합하면, 보완은 **세 층위**로 나뉜다. 체감 효과가 큰 순서.

### 🥇 1순위 — Apple Vision OCR 브리지 구현 (iOS 체감 최대)
현재 스텁으로 막힌 앱의 **원래 주력 경로**를 연결하는 것이 단일 최대 개선이다.

- Apple Vision(`VNRecognizeTextRequest`)은 **기울기/회전 자동 보정 · 다중 방향 · 한글 인쇄체**에 강하여, 본 테스트의 3대 실패(R04 회전, R01·R05 검출 누락)를 사실상 해소할 것으로 기대.
- **온디바이스 · 무료 · 사용자 동의 불필요 · 네트워크 불필요** (CLOVA 대비 이점).
- 산출물이 `lines + boundingBox + confidence` 형태라 기존 `ReceiptRecognizerResult` 계약에 그대로 부합 → 통합 비용 낮음.
- 완료 시 `IOS_INTERIM_PPOCR_FALLBACK` 를 끄고 Apple Vision을 실주력으로 승격.

### 🥈 2순위 — 공용 PP-OCR 폴백 강화 (Android 주력 + iOS 폴백)
PP-OCR은 **Android에서는 여전히 주력**이고 iOS 폴백이므로 개선 가치가 있다. 오늘 관찰된 결함 순.

| 결함 (관찰 근거) | 보완 |
|---|---|
| **기울기 보정 없음** — 축 정렬 박스·회전 crop 없음 (R04 붕괴) | 연결요소를 **최소외접 회전사각형**으로 잡고 **perspective/rotation 워프 crop** |
| **단일 960 스케일 검출** — 큰 영수증 본문 통째 누락 (R01·R05) | 검출 입력 해상도 상향(1280~1536) 또는 타일링·다중 스케일 |
| **가로행 병합 과함** — 서로 다른 표 칸이 한 crop에 혼입 | `mergeTextRows` 완화 + **열(칸) 인지 분할**, 폭 320 초과분 truncation 방지 |
| **신뢰도 = 로짓 평균**(softmax 아님) | softmax 확률 평균으로 교체 후 0.35 임계값 재보정 |
| 방향(0/180) 분류기 없음 | 텍스트 방향 분류 단계 추가 |

> 이 개선들은 **오늘의 8장으로 before/after 재측정**이 가능하다(동일 러너 보유).

### 🥉 3순위 — 도메인 특화 (모든 엔진에 공통 이득)
- **필드 앵커 추출:** 인쇄 라벨(품명·수량·TEL·배달장소·보내는분)은 오늘도 신뢰도 0.9~1.0으로 안정적. 이 라벨을 앵커로 **기하학적으로 인접한 값**을 뽑으면 인식이 불완전해도 실사용 **필드 정확도**가 오른다. (`app/ocr/fieldRegistry.ts`·`layout.ts` 골격 확장.)
- **촬영 가이드 UX:** 캡처 시 기울기/프레이밍 힌트로 나쁜 입력 자체를 감소.
- **CLOVA 폴백 구현:** 저신뢰 영수증 재시도용(동의·네트워크·비용 수반) — 기본이 아닌 안전망.

### 추천 순서
1. **Apple Vision 브리지** (iOS 실사용 즉효) →
2. **PP-OCR deskew + 다중스케일** (Android·폴백, 8장 before/after 재측정) →
3. **필드 앵커링** (정확도 상한 상승).
