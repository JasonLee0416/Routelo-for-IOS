## 2. 방법

앱의 실제 파이프라인(`recognizeReceiptWithPpOcr`)과 **동일한 로직**을 Node에서 재현하여 각 이미지를 인식했다.

1. **검출 전처리** — 원본을 최대변 960px로 축소(32 배수 정렬), ImageNet 정규화, NCHW float32 텐서.
2. **텍스트 검출** — DB(Differentiable Binarization) 후처리: 확률 임계값 `0.3`, boxThreshold `0.5`, minArea `12`, unclip `1.6`, 연결요소 → 축 정렬 박스 → 가로행 병합(수직 겹침 ≥0.45).
3. **인식 전처리** — 각 박스를 원본에서 crop, 높이 48·폭≤320으로 리사이즈, `/127.5 − 1` 정규화.
4. **텍스트 인식** — 인식 모델 추론 후 **CTC 그리디 디코딩**(사전 매핑), 신뢰도 = 스텝별 최대 로짓 평균.
5. **필터** — 신뢰도 `≥ 0.35` 인 줄만 채택.

**유일한 근사:** 이미지 리사이즈/크롭이 앱에서는 iOS 네이티브(`expo-image-manipulator`), 본 테스트에서는 bilinear로 구현됨. 인식 결과에 미치는 영향은 미미.

## 5. 핵심 발견

1. **검출 안정성 편차가 큼.** 모델 자체의 한글 인식력은 준수하나(R07은 26줄 중 전화·상품명·배달장소·받는분·보내는분까지 대부분 포착), 검출 단계가 영수증마다 불안정하여 R01·R04·R05에서는 본문 대부분을 놓쳤다.
2. **기울기 보정(deskew)이 없음.** 검출 박스가 **축 정렬(axis-aligned)** 이고 회전 crop이 없어, R04처럼 크게 기울어진 영수증은 글자가 사선으로 잘려 인식이 붕괴한다. 실사용에서 기사님이 비스듬히 촬영하면 그대로 실패로 이어진다.
3. **가로행 병합이 과함.** 한 가로줄을 넓게 묶어(수직 겹침 기준) 서로 다른 표 칸(예: “품명 / 축하3단 / 수량 / 1”)이 하나의 crop에 섞여 들어가 오인식을 유발한다.
4. **선명해도 놓치는 케이스(R01).** 사람 눈에는 매우 선명한데도 검출이 4영역만 잡고 로고 그래픽을 오독했다 — 단일 960 스케일 + DB 임계값 조합의 한계로 보인다.
5. **표 라벨(품명·수량·TEL·배달장소 등) 인식은 강함.** 고정 인쇄 라벨은 신뢰도 0.9~1.0으로 안정적 → 필드 앵커로 활용 여지.

## 6. 개선 제안 (우선순위)

1. **각도 보정 + 회전 crop** — 검출 다각형 기반 최소외접사각형으로 crop을 워프(perspective/rotation)하여 사선 텍스트 대응. (R04·R01 개선 기대)
2. **다중 스케일 / 상향 해상도 검출** — 960 단일 스케일 대신 큰 영수증에 대해 스케일 상향 또는 타일링.
3. **행 병합 완화 / 열 분할** — 표 격자를 인지해 칸 단위로 crop 분리.
4. **신뢰도 지표 개선** — 로짓 평균 대신 softmax 확률 평균으로 바꿔 임계값 튜닝을 직관화.

## 7. 재현 방법

앱과 동일 파이프라인을 Node에서 재현:

```bash
cd routelo
ONNXRUNTIME_NODE_INSTALL=skip npm install   # onnxruntime-node CPU 바인딩은 번들됨(CUDA 불필요)
npx patch-package                            # --ignore-scripts로 설치했을 경우
# 검출/인식/CTC 로직: app/ocr/ppocr/{image,dbPostprocess,ctc}.ts 를 Node 러너로 포팅해 실행
```

- 모델·사전 자산: `assets/ocr/` (manifest `assets/ocr/manifest.json`).
- 임계값·정규화·CTC 파라미터는 위 2절 그대로.
